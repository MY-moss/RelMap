### 1.2 竞品核心不足 (RelMap 可切入的机会)

**Dex 的短板:**
- 强依赖 Gmail/日历生态，非 Google 用户难以使用
- 定价偏高，个人用户承受力有限
- 关系图谱缺失，无法可视化社交网络
- 所有数据在云端，隐私零保障

**Clay 的短板:**
- 强依赖 LinkedIn/Twitter 自动化补全，国内无法使用
- 功能过于轻量，无法作为"关系管理"中心
- 无关系量化评分
- 无本地化，中文支持差

**Monica 的短板:**
- 强依赖浏览器扩展，脱离浏览器无使用价值
- 关系管理粒度粗（仅有标签）
- 无亲密度量化
- 免费版功能受限严重

**Moment 的短板:**
- 仅提醒"该联系了"，缺乏深度管理能力
- 手动录入繁琐，无自动导入
- 无社交图谱
- 数据不可导出

**国内工具的普遍短板:**
- 商业模式驱动，隐私让位于广告/推荐
- 关系管理停留在通讯录增强层面
- 无 AI 分析能力
- 数据难以导出/迁移

## 2. RelMap 的差异化定位

### 2.1 核心优势总结

```
纯本地     +     开源     +     AI 能力
(隐私安全)     (可控可扩展)     (智能增强)
```

这是 RelMap 的"不可能三角"——竞品中没有任何一个能同时兑现这三者。

### 2.2 产品定位声明

> **RelMap 是你的人际关系的私人地图——纯本地运行、AI 辅助、开源透明的第二代关系管理工具，让每个人都能安全地理解并维护自己的社交网络。**

### 2.3 用户群体画像

**核心用户 (T1): 隐私敏感的知识工作者**
- 年龄 25-45 岁，科技/金融/法律/医疗行业
- 对数据隐私有高度意识，不愿将社交数据交给第三方
- 需要管理 50-500 人的社交网络
- 习惯桌面端工作，愿意自助配置 (BYOK)
- 代表: 创业者、投资人、销售总监、自由职业者

**次核心用户 (T2): 数字极简主义倡导者**
- 寻求替代 Dex/Clay 的更私密方案
- 追求 "less cloud, more control"
- 愿意为隐私付出学习成本
- 代表: 开源爱好者、隐私倡导者、独立开发者

**潜力用户 (T3): 重度社交的管理者**
- 商务人士，需要记忆大量人脉关系细节
- 通过关系图谱发现潜在的社交机会
- 代表: 商务拓展、HR、猎头、市场营销

### 2.4 "不做"的功能边界（产品哲学）

1. **不做云端同步**——数据永远在你的设备上。如需同步，用户自行通过 Git/Nextcloud 等工具管理
2. **不做社交网络爬虫**——不抓取 LinkedIn/Twitter/微信数据，手动录入或 vCard/CSV 导入
3. **不做 SaaS 订阅**——免费开源，仅让用户自带 AI Key (BYOK)
4. **不做移动端优先**——桌面端是第一公民，移动端作为延伸
5. **不做商业变现**——不出售数据、不展示广告、不强制付费

### 2.5 与竞品的关键差异点 (5+)

| # | 差异点 | RelMap | 竞品 |
|---|--------|--------|------|
| 1 | **数据主权** | 纯本地 SQLite，用户 100% 拥有全部数据 | 云端存储，平台可访问用户社交数据 |
| 2 | **关系图谱可视化** | Cytoscape 力导向图，亲密度颜色/大小编码 | 列表视图为主，无关系网络 |
| 3 | **亲密度量化体系** | 四维算法 (频率+近因+深度+手动) 综合评分 | 仅有/无标签，缺乏量化 |
| 4 | **AI 灵活性 (BYOK)** | 自带 API Key，可选 OpenAI/Claude/Ollama 等 | 固定 AI 服务，用户无选择权 |
| 5 | **开源透明** | 完整源码，可审计、可 Fork、可贡献 | 闭源，黑盒算法 |
| 6 | **离线可用** | 断网情况下全部功能可用 | 绝大多数功能依赖网络 |
| 7 | **全面性** | 人物+社交+关系+事件+日记+相片+交互+提醒 一站式 | 单一维度(仅同步/仅提醒/仅标签) |

## 3. 可参考的"新奇"功能灵感

### 3.1 Notion / Obsidian → "社交笔记"思路

关系管理不应与知识分离。借鉴:
- **双向链接**: 每个联系人的页面自动关联事件、日记、照片
- **关系图谱与知识图谱融合**: 让关系图谱中不仅有人物，还有共同事件/项目节点
- **Markdown 笔记**: 联系人页支持富笔记，记录交往细节

### 3.2 Spotify Wrapped → "社交年度报告"

- **年度关系回顾**: "今年你与 X 位朋友保持联系，最亲密的人是 X，你们共度了 X 个事件"
- **季度/月度简报**: 关系健康度变化趋势
- **断联排行榜**: "你快要失去联系的朋友有..." (目前有断联检测，可增加数据叙事)

### 3.3 Duolingo → 游戏化日常参与

- **每日关系维护打卡**: "今天联系一个朋友吧！" 连续打卡奖励
- **关系经验值 (XP)**: 维护关系可获得经验值，升级解锁新功能
- **成就徽章**: "社交达人"(联系 100 人)、"温暖如初"(联系持续 1 年)、"人脉王"(500+ 联系人)
- **排行榜(可选)**: 匿名比较"关系维护指数"

### 3.4 GitHub 贡献图 → 社交热力图

- **联系日历**: 类似 GitHub 绿色方块图，展示每天与多少人互动
- **强度可视化**: 颜色深浅映射交互密度
- **模式发现**: 发现自己的社交规律 (比如每周五见朋友最多)

### 3.5 Roam Research / Logseq → 时间线 + 双向引用

- **自动时间线**: 所有与某个人的交互自动汇总为时间线
- **上下文记忆**: "2024 年 3 月和张三一起吃火锅" 自动关联到张三的页面
- **图谱搜索**: 在关系图谱中点选节点，展现该人的完整"社交宇宙"

### 3.6 Apple Screen Time → "温柔"的数据展示方式

- **不制造焦虑**: 提醒 "你已 X 天未联系某人"，但以温和语气而非警告
- **正面激励**: "联系老朋友让你更快乐，本周你联系了 5 位朋友" 
- **数据叙事**: 用图表讲故事，而非冰冷数字 (当前 RelMap 已有仪表盘雏形)

## 4. 功能优先级建议

### 4.1 "必须有"的基础功能 (P0)

这些是竞品已有、用户预期 RelMap 也应具备的能力:

| 功能 | 优先级 | 说明 | 关联竞品 |
|------|--------|------|---------|
| 搜索增强 | P0 | 当前已有 FTS5，但缺少全局搜索 UI | 所有竞品 |
| 批量导入 (手机号+Email) | P0 | 从通讯录/CSV 批量导入无障碍 | Dex/Clay |
| 联系人合并去重 | P0 | 重复联系人检测与合并 | 所有竞品 |
| 分组/标签筛选 | P0 | 已有基础实现，需优化交互 | 所有竞品 |
| 数据导出可读格式 | P0 | 已有 JSON/CSV，可增强 HTML 报告 | 所有竞品 |
| 联系频率设置 | P0 | 为不同人设置期望联系频率 | Moment/Dex |

### 4.2 "人无我有"的杀手功能 (P1)

基于 RelMap 的纯本地 + AI + 开源优势打造:

| 功能 | 独特性 | 实现路径 |
|------|--------|---------|
| **关系图谱智能分析** | 利用图算法(PageRank/社群发现)自动识别"关键联系人"、"社交圈子" | Cytoscape + 后端分析 |
| **BYOK AI 功能矩阵** | OCR 增强、日记分析、语义搜索、联系时机建议 | BYOK 架构已设计完成 |
| **亲密度趋势图** | 可视化与每个联系人关系的变化曲线 | 已有 IntimacyScore，缺可视化 |
| **本地语义搜索** | 用 Embedding 替代关键词，自然语言检索 | BYOK + FTS5 混合 |
| **联系节奏分析** | "你平均每 X 天联系一次张三，建议加快频率" | 交互日志 + AI 分析 |
| **社交网络洞察** | "李四和王五都与你是好友，但他们互不相识" | 关系图谱 + 社群发现 |
| **私密 AI 日记助手** | AI 分析日记提取人物/情绪/关键词，自动建立关联 | 已有 Diary + BYOK AI |

### 4.3 "锦上添花"的亮点功能 (P2)

新奇感 + 病毒传播性:

| 功能 | 灵感来源 | 传播点 |
|------|---------|--------|
| **年度关系报告 (RelMap Wrapped)** | Spotify Wrapped | 可导出分享截图 |
| **社交热力图** | GitHub 贡献图 | 视觉冲击，用户自发分享 |
| **关系 AI 对话** | "帮我梳理我和张三四年来的关系变化" | 可录制成短视频传播 |
| **游戏化成就** | Duolingo | 成就徽章可分享 |
| **关系健康度仪表盘** | Apple Screen Time | 温和提醒，培养习惯 |
| **开源社区贡献图谱** | GitHub | 社区成员在 RelMap 图谱中可见 |
| **数据故事模式** | 叙事化展示个人社交故事 | 年终总结类传播 |

## 5. 产品演进路径建议

### Phase 1: 核心体验打磨 (当前 → 3 个月)

**目标**: 达到"可用"标准，对标竞品基础功能，不流失用户

**功能**: 
- P0 功能全部完成
- BYOK AI 基础能力上线 (OCR 增强 + 日记分析)
- 搜索 UI 优化 (全局搜索 + 过滤)
- 联系人合并去重
- 交互体验打磨 (空状态、加载态、错误处理)

**预期用户增长**: 0 → 500 (种子用户，通过 GitHub/开源社区获取)

**口碑传播点**: "一个用心做关系管理的开源项目"

### Phase 2: 差异化功能建设 (3-6 个月)

**目标**: 建立"本地 AI 关系管理"的品牌认知，形成竞争壁垒

**功能**:
- P1 杀手功能分批上线
- 关系图谱智能分析 (社群发现 + 关键人识别)
- 亲密度趋势图 + 联系节奏分析
- 语义搜索 (BYOK Embedding)
- 社交网络洞察
- 隐私白皮书 + 安全审计

**预期用户增长**: 500 → 5,000 (科技圈口碑传播)

**口碑传播点**: "原来关系管理可以这么智能又安全"

### Phase 3: 生态扩展 (6-12 个月)

**目标**: 从工具发展为平台/生态

**功能**:
- P2 亮点功能上线
- 插件系统 (社区可开发自定义分析插件)
- 移动端 Companion App (轻量级查看 + 提醒)
- Web 版 (用户自行托管)
- 社区模板库 (关系管理最佳实践分享)
- 企业版 (团队关系管理)

**预期用户增长**: 5,000 → 50,000+ (跨出技术圈，进入主流)

**口碑传播点**: "开源界的 Notion for relationships"

## 6. 战略总结

### RelMap 的竞争护城河

```
第一层 (最外层): 开源透明
  → 代码可审计、可 Fork，建立信任基础

第二层: 纯本地 + BYOK AI
  → 数据主权 + AI 灵活性，竞品无法复制（商业模式约束）

第三层 (最内层): 关系图谱 + 亲密度量化
  → 独家数据模型，用户使用越久，切换成本越高
```

### 当前应立即行动的三件事

1. **完成 BYOK 架构实施** (已有完整设计方案) —— 这是 AI 差异化能力的基石
2. **优化搜索与批量导入体验** —— 用户上手的第一触点，决定留存
3. **启动开源社区建设** —— 写清晰的 CONTRIBUTING.md，发布 Roadmap，吸引贡献者
