## 一、项目定位

一个**本地运行的AI增强型个人关系管理应用**，以001方案为唯一技术基础框架，经多智能体协同研究后整合002/003/004方案中可通过AI实现的优质功能模块，专为全流程AI托管开发设计。

### 技术栈锁定（不可变更）

| 层级 | 技术 | 版本 | AI生成率 |
|------|------|------|:--------:|
| 桌面框架 | Electron | ^28.0 | 85% |
| 前端框架 | React + TypeScript | ^18.3 / ^5.4 | 92% |
| 样式框架 | TailwindCSS | ^3.4 | 88% |
| 状态管理 | Zustand | ^4.5 | 80% |
| 本地数据库 | better-sqlite3 | ^11.0 | 95% |
| 全文搜索 | SQLite FTS5 | 内置 | 90% |
| 人脸识别 | face-api.js | ^0.22 | 90% |
| OCR引擎 | Tesseract.js | ^5.0 | 85% |
| 关系图谱 | Cytoscape.js | ^3.28 | 70% |
| 数据加密 | better-sqlite3加密 + AES-256 | - | 80% |

### 明确排除的技术（与001方向冲突，已剔除）

| 技术 | 来源方案 | 排除理由 |
|------|---------|---------|
| Tauri / Rust | 002 | AI写Rust编译错误无法自修复 |
| ONNX Runtime | 002 | AI训练数据少，容易出错 |
| Flutter / Dart | 004 | AI对Flutter不熟悉 |
| Syncthing同步 | 004 | AI没见过这种代码，且违背零联网原则 |
| 云端AI API | 004 | 需要API Key管理，增加复杂度，违背本地优先 |
| Python后端 | 003 | 三语言割裂，AI容易混乱 |

## 五、快速开始

### 第一步：环境准备

```bash
cd relmap
npm install
```

### 第二步：开发

```bash
npm run dev      # 启动开发服务器
npm run build    # 构建生产包
npm run test     # 运行测试
npm run lint     # 代码检查
```

### 第三步：AI开发操作守则

遵循 `项目方案/02-AI能力方案/02-AI开发方法论.md`：

1. 一次只让AI做一个文件（每文件≤200行）
2. 给AI"正例"而不是"描述"（贴已有代码让AI模仿）
3. 每次只改一个需求（改完确认再下一个）
4. 关键模块先写测试
5. 每步git commit

## 六、多智能体协同研究机制

本方案由以下7个智能体角色协同完成研究和整理：

| 角色 | 职责 | 本次贡献 |
|------|------|---------|
| 产品经理(PM) | 产品规划、用户需求 | 确认功能优先级和用户价值评估 |
| 技术架构师(SA) | 系统架构、技术选型 | 锁定001方案技术栈，审查内容一致性 |
| AI工程专家(AIA) | AI模型集成、LLM开发 | 评估AI代码生成率，确认face-api.js/Tesseract.js选型 |
| 数据架构师(DA) | 数据模型设计 | 设计14张核心表+FTS5，审批跨方案数据模型 |
| 安全专家 | 信息安全、数据保护 | 确认零联网、本地加密、PIPL合规方案 |
| 用户体验设计师 | 交互设计、可用性 | 设计7个主页面和组件规范 |
| 项目经理(PMGR) | 项目管理、进度控制 | 制定开发路线图和Phase验证清单 |

详细讨论过程见 `项目方案/历史归档/多智能体会议记录.md`。
